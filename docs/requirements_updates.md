The platform section describes Google Apps Script + Google Sheets as the technology stack. This is now outdated. The tool was rebuilt on:

- **Database:** Supabase (PostgreSQL)
- **Frontend:** Vanilla HTML/JS, hosted as static files
- **Hosting:** GitHub Pages (or Cloudflare Pages)
- **Storage:** Supabase Storage (for photos)
- **Auth:** Supabase Auth (email/password)

Update the platform table and rationale. The client-ownership requirement is satisfied: the Supabase project can be transferred to a Faith Lock account and the static files live in a GitHub repo the client can own.

### Auth Model (Section 4.1)

The PIN-based admin access model described in R-02 and R-03 has been replaced with Supabase email/password authentication and a role-based profile table. Update:

- R-02: Admin and tech roles are now determined by the `user_profiles` table linked to Supabase Auth, not a PIN or name dropdown
- R-03: The admin PIN requirement is replaced by standard password-based auth. Pre-go-live requirement is now: change the default passwords, not a PIN
- Add new requirement: each tech should have their own Supabase Auth account before go-live. The POC uses a shared tech login which is a known limitation

### New Features Built (Section 4, various)

The following requirements were built during the POC that do not exist in v2.0 and should be added:

**Expected Parts and Labor on Job Assignment**
- Admin can pre-load expected parts (with quantities) and expected labor (type and hours) when assigning a job
- These appear on the tech's job card as a blue informational banner
- They are pre-populated in the completion form and editable by the tech
- On submission, the tech's actuals replace the expected items in the line items table

**Unsubmit**
- A tech can remove their own submission if the job has not yet been reviewed by admin
- The job returns to Scheduled status
- The completion record and line items are cleared
- The tech can resubmit with corrected information

**Account Instructions**
- Each account has an Instructions tab in the detail view
- Free-text field for gate codes, contacts, access restrictions, billing quirks
- Visible to admin only (not surfaced in tech view in current build)
- Note: currently stored in browser localStorage (POC limitation). Production build requires a dedicated database column

**Calendar View**
- Weekly calendar showing assigned jobs per tech
- Navigable by week (previous/next)
- Tech selector dropdown; defaults to all techs
- Jobs color-coded by status
- Multi-tech jobs display lead designation

**Reference Data Full CRUD**
- All reference tables (Parts, Job Types, Labor Types, Techs) now support inline editing
- Any field can be edited directly in the table row and saved without navigating away
- Add, edit, and deactivate all record types

**Account CSV Export**
- Two export functions on the Accounts page:
  - Full filtered account list to CSV (name, type, parent, address, phone, billing notes, job count, last service, active)
  - Per-account job history to CSV (all jobs with date, type, status, payment, total, tech notes, scope)

### Data Model Updates (Section 5)

The data model section describes Google Sheets tabs. Replace with Supabase tables. The logical structure is the same but the implementation is a relational PostgreSQL database. Key differences:

- All IDs are UUIDs generated by the database
- `assigned_tech_ids` is a UUID array column on the Jobs table (stores Supabase Auth UUIDs in production; techs-table UUIDs in POC - this is a known inconsistency to resolve before go-live)
- `job_completions` is updated in place by admin reopen/edit (not appended as a new row)
- Photos are stored in Supabase Storage; the `job_photos` table stores the storage path and metadata
- Account instructions are currently in browser localStorage (POC only); production requires an `instructions` text column on the `accounts` table
